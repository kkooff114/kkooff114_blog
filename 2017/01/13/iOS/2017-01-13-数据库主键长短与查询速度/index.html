问题

之前在设计移动请求缓存的时候碰到这样一个问题:

将每次请求后台api返回的数据进行缓存, 下次再次请求同样api的时候, 直接从缓存中读取结果, 无需发送请求. 于是, 在每次有效的发送请求后, 将url+request的string值作为key, response作为value, 存入一张表中:

key(url+requestParams) value(response)
string string

以后每次请求的时候先去这张表中按key查找, 如果命中, 则直接返回, 如果没有命中, 则发送请求到后台.

此时就有个疑问, key如果很长, 是不是会对查询的效率造成影响, 于是便有了下面的测试.
